# MCP 输入输出调试日志 — 实现计划与风险评估结论

## 1. 目标与实现要点（原计划摘要）

- **目标**：在所有 MCP 工具调用处统一打印输入/输出日志，仅当 `TEAM_MEMORY_DEBUG=1` 或 `TEAM_MEMORY_MCP_DEBUG=1` 时生效；单条内容超过 300 字符时控制台只打前 300 字，完整内容写入临时文件并打印路径；临时目录下最多保留 20 个此类文件。
- **实现**：新增 `mcp_debug_log.py`（debug 开关、300 字截断、落盘、20 文件清理）；在 `server.py` 的 `track_usage` 装饰器内调用。
- **涉及文件**：新建 `src/team_memory/mcp_debug_log.py`，修改 `src/team_memory/server.py`。

## 2. 风险评估与缓解（已与用户确认）

以下为子 Agent 评估后的风险点及用户确认结果。

### 2.1 R1.2 并发隔离（用户指定方案）

- **风险**：多请求并发写同一目录的竞态。
- **确认方案**：**用 Chat 名做隔离**，减少冲突（即按会话/Conversation 维度分目录或前缀）；**同一 Chat 内**若仍可能冲突，则文件名带**时间戳 + 随机/UUID**。
- **实现注意**：若 MCP 请求上下文中可获取 Chat/session/conversation 标识（如 Cursor 传入的 session id 或 thread id），则以其作为落盘子目录名或文件名前缀；否则退化为单目录 + 时间戳+UUID 文件名。同一 Chat 内文件名格式：`{tool}_{timestamp}_{short_uuid}_in.txt` / `_out.txt`。

### 2.2 其余风险 — 全部按评估建议采纳

| 编号 | 风险点 | 采纳的缓解措施 |
|------|--------|----------------|
| R1.1 | 同步 IO 拉长响应 | debug 时落盘；写文件用 asyncio.to_thread 或线程池；或提供「仅控制台不落盘」选项。 |
| R1.3 | 大结果序列化 CPU/内存 | 先长度判断，超阈值再序列化；或只序列化/截断前 N 字符。 |
| R2.1 | 敏感入参 | 敏感参数名列表，序列化前脱敏（*** 或仅长度），文档说明。 |
| R2.2 | 敏感返回结果 | 对已知敏感字段脱敏；或仅记录结果长度+是否成功；或配置关闭结果落盘。 |
| R2.3 | 临时文件权限与清理 | 落盘 0o600；文档写明仅调试用；可选启动时清理 >24h 旧文件。 |
| R2.4 | 生产误开 debug | 文档与部署清单明确生产禁止；可选非开发环境 debug=1 时 warning 或拒绝落盘。 |
| R3.1 | 与 track_usage/Hook 重叠 | 明确 MCP 调试仅本地排查；与 HookRegistry 用途区分。 |
| R3.2 | logger 体系 | 使用 `logging.getLogger("team_memory.mcp_debug")` 或带固定前缀。 |
| R3.3 | 双环境变量语义 | 文档明确仅 MCP_DEBUG 控制 MCP 输入输出；代码注释写清。 |
| R4.1 | .debug 不可写 | 落盘前检查可写；不可写则降级为仅控制台截断 + warning。 |
| R4.2 | .debug 相对路径歧义 | 优先项目根/配置 base_dir 解析；否则 tempfile.gettempdir() 子目录并打出路径。 |
| R4.3 | 容器 temp 易丢 | 文档说明容器内可设环境变量指定落盘目录。 |
| R5.1 | 序列化失败影响调用 | 记录逻辑独立 try/except，仅 logger.warning，不改变异常传播；序列化用安全方式。 |
| R5.2 | 磁盘满/权限错误 | 记录逻辑整体 try/except，失败只打日志不 re-raise。 |
| R5.3 | 20 文件清理边界 | 清理逻辑 try/except；明确「最多保留 20 个」语义；单测覆盖。 |

## 3. 实现时需落实的要点

- **R1.2**：落盘路径或文件名体现「Chat 隔离」；同 Chat 内用 `时间戳 + 随机/UUID` 避免重名。
- **20 文件上限**：按「每个 Chat 子目录内最多 20 个」或「全局最多 20 个」在计划中二选一（建议按 Chat 子目录各 20，避免单 Chat 写爆）。
- **敏感字段**：实现时在 `mcp_debug_log` 中维护敏感参数名列表并对输入/输出做脱敏。
- **异常隔离**：所有记录输入/输出的代码包在独立 try/except，不影响 `track_usage` 的原有异常传播。
