# 开发者工作流与项目进度管理 — 三视角 PK 后的最终三方案

> 基于三位 Agent 分别从**技术实用程度**、**使用者角度**、**项目助力角度**的分析与候选方案，进行交叉 PK 后选出的 **3 个最终最优秀方案**。
> **约束**：所有方案均与 team_memory 现有 Task 能力（tm_task、tm_task_claim、tm_ready、tm_dependency、tm_message、完成即沉淀经验、Web 看板、Git [TM-xxx]）深度结合。

---

## 〇、补充约束与演进方向（必读）

### Case 1：企业内网 / 无 GitHub — 当前版本不以 Issue 为任务源

- **TM 可能在企业的内网环境使用**，无法或不允许依赖 GitHub Issues/Projects 做任务管理。
- **当前版本暂不考虑**以 GitHub Issue（或其它外部 Issue 系统）作为任务管理源或双向同步；任务创建、认领、就绪、完成**全部在 TM 内**完成。
- 因此：**方案一（TM 为唯一任务源）** 与 **方案三（任务组完成时按总-分或总-分-分沉淀，紧相关不归类）** 为**当前版本主推**，二者组合使用；依赖外部 Issue/工单同步的「方案二」仅作为**未来扩展**，不在本期实现范围内。

### Case 2：演进方向 — 架构管理 + 经验挂载到架构图

- **后续演进**：TM 将引入**项目架构管理**能力（类似 GitNexus）：对整体架构提供**可视化**（文件/函数/类/调用关系、模块聚类、执行流等）。
- **经验挂载**：
  - **任务完成后**：将任务关联的经验**同步到架构图**，挂到本次改动涉及的架构节点（如改动的文件、函数、模块）上。
  - **后续改动某部分代码/架构时**：根据当前编辑或 diff 的架构位置，**自动或推荐**把相关经验挂到对应节点，使检索与推荐按「架构位置」生效，**提升项目经验沉淀的质量**（按位置查、按影响面推）。
- 与当前方案的关系：方案一/三的「完成即沉淀」在设计上**预留「影响的架构节点」**（如本次任务涉及的 file/symbol 列表），便于未来「任务完成 → 同步到架构图」与「改动代码 → 经验挂到对应位置」的闭环；工作流与数据模型可预先考虑「经验 ↔ 架构节点」的关联字段或索引。

## 一、三视角输入与 PK 结论摘要

| 视角 | 分析的 3+ 种工作流示例 | 推荐的 3 个候选方案 | PK 中采纳 |
|------|------------------------|----------------------|-----------|
| **技术实用程度** | CCPM、GitHub Projects+Sub-issues、Agentic PM/LEO、ZenHub/Actions | A: tm 唯一任务源；B: GitHub 为主+tm 执行；C: CCPM worktrees+tm | A→方案一；B→方案二；C 作为方案二的技术增强或可选扩展 |
| **使用者角度** | CCPM 风格、GitHub Projects+Issues、ZenHub/Pipeline | 1: TM 唯一+Cursor/Web 双端；2: TM 主+GitHub 镜像；3: GitHub 主+TM 清单 | 1→方案一；2→方案二；3 作为方案二的用户侧折中 |
| **项目助力角度** | GitHub Issues/Projects、ZenHub/Linear、CCPM/APM | A: TM 复盘中枢；B: TM 看板+GitHub 同步；C: 任务组=交付单元+经验组 | B→方案二；C→方案三；A 作为方案三的轻量形态或补充 |

**PK 规则**：
- 三个视角都强烈重合的「TM 为唯一任务源」→ **方案一**（**当前版本主推**）。
- 「TM 看板 + 外部 Issue/工单同步」在部分场景被推荐 → **方案二**；因 **Case 1（企业内网/无 GitHub）**，**当前版本不实现**，仅作未来扩展描述。
- 「任务组 = 交付单元 + 整组沉淀为经验组」在项目助力上独特且与 tm 能力完全契合 → **方案三**（**当前版本主推**）。
- CCPM worktrees + tm 作为**可选扩展**，可与方案一/三叠加，不依赖 GitHub。

## 二、最终方案一：TM 为唯一任务源（Cursor + Web 双端）

### 2.1 方案描述

- 任务**只**在 team_memory 创建与推进（tm_task create/list/update）；GitHub 仅作代码与 PR 载体，不做任务主库。
- 选活用 **tm_ready**，认领用 **tm_task_claim**，执行中用 **tm_message** 留记录，完成用 **tm_task(update, status=completed, summary=...)**，自动沉淀经验。
- 进度与打断/恢复全在 TM：Web 看板 + 分组 + 「进行中」列 + 任务留言。

### 2.2 与 team_memory Task 的结合方式

| 能力 | 用法 |
|------|------|
| **tm_task** | create 建任务；list/get 看板与详情；update(completed, summary) 完成并沉淀经验 |
| **tm_task_claim** | 认领后 in_progress，遵守 WIP 限制 |
| **tm_ready** | 「接下来做什么」的入口，只显示依赖已满足的任务 |
| **tm_dependency** | 管理 blocks/related，决定 tm_ready 结果 |
| **tm_message** | 打断后恢复的上下文，与 Web 任务详情时间线一致 |
| **Web 看板** | 按 project/分组筛选，五列 Kanban，任务组进度环 |
| **Git [TM-xxx]** | commit 中带任务 ID，便于代码与任务追溯 |

### 2.3 技术优势

- **落地成本最低**：无需 GitH Issue 同步、双写或冲突处理；与现有 MCP 契约完全一致。
- **可扩展性高**：依赖、多项目、分组、WIP 均已在 tm 内支持；后续加字段或策略不改外部系统。
- **就绪与认领语义清晰**：tm_ready + tm_task_claim 即「可执行任务 + 原子认领」，无跨系统状态不一致。

### 2.4 收益

- **可监督**：Web 看板 + tm_task list 即进度；所有状态单源。
- **可打断 / 可重启**：in_progress + tm_message 即恢复点；无需记 worktree 或 Issue 对应关系。
- **可重复**：同一套流程可反复跑（创建任务 → 认领 → 完成 → 沉淀），适合迭代与模版化。
- **完成即知识**：每条 completed + summary 自动成经验，tm_search/tm_solve 可检索。

### 2.5 适用场景

- 以 Cursor/Agent 与 TM Web 为主的团队；不强制要求「任务必须在 GitHub 上可见」。
- 小团队或个人希望**心智负担小、一个地方管任务与经验**。

### 2.6 与方案三的组合（推荐）

- **当前版本推荐**：方案一与方案三**一起使用**：日常单任务按方案一（TM 唯一任务源、完成即单经验）；当存在**任务组**时，采用方案三的**总-分**（组内任务紧相关）或 **总-分-分**（组内类型混杂时归类）沉淀策略（见第四节），既保留单经验独立性，又通过「总」建立适度关联。

## 三、方案二（未来扩展）：TM 看板 + 外部工单/Issue 对接 — 当前版本不实现

> **因 Case 1**：企业内网或无法使用 GitHub 的场景下，当前版本**不以任何外部 Issue 系统为任务源**；本节仅描述未来若支持「与外部工单/Issue 系统对接」时的形态，**不纳入本期实现范围**。

### 3.1 方案描述（未来形态）

- 以 **team_memory 看板**为 Agent/人工统一操作入口；关键任务与 **外部工单/Issue**（如企业内 Jira、GitLab Issue、自建工单）一一绑定。
- 在 TM 创建/更新任务时，通过适配器 **同步到外部系统**（创建/更新工单、状态、评论）；完成时 **tm_task(update, completed, summary)** 自动沉淀经验，并**回写工单**（关闭 + 评论，可选带经验 ID）。
- 可选：commit 含 `[TM-xxx]`，形成「需求→任务→代码→经验」全链路；**不依赖 GitHub**，由适配器对接企业现有系统。

### 3.2 与 team_memory Task 的结合方式（未来）

| 能力 | 用法 |
|------|------|
| **tm_task** | create 时可选「同步到外部工单」→ 创建工单；update(completed, summary) → 沉淀经验 + 回写工单状态/评论 |
| **tm_task_claim** | 认领在 TM；可选通过适配器把工单标为 In Progress |
| **tm_ready** | 只显示 TM 内依赖已满足的任务；若任务来自工单同步，需在同步时处理好父子/阻塞关系 |
| **tm_dependency** | 在 TM 内管理；与外部系统的层级可做单向或双向映射（建议以 TM 为源） |
| **Web 看板** | 主战场；外部系统为可选镜像，满足对外/合规「在工单系统上能看到」 |

### 3.3 技术优势

- **单一操作面**：日常在 TM 看板与 Cursor 内完成选活、认领、完成；GitHub 状态由同步逻辑维护，减少人工双写。
- **可追溯**：Issue 编号 + TM 任务 ID + 经验 ID 可互相引用；PR/commit 与 Issue 的既有能力保留。
- **与现有 Task 能力兼容**：完成即沉淀仍由 tm_task(update, completed, summary) 触发；仅增加「回写 Issue」的钩子或后台任务。

### 3.4 收益（未来）

- **可监督**：TM 看板 + 外部工单列表双重可见；适合需要对外汇报或合规留痕的团队。
- **可打断 / 可重启**：以 TM 的 status + tm_message 为主；恢复时看 TM，工单评论可作补充。
- **完成即知识 + 可追溯**：经验自动落库；工单关闭与评论提供审计线索。

### 3.5 适用场景（未来）

- 企业已有 **Jira/GitLab/自建工单** 等，必须保留为对外或合规入口，同时希望 Agent 与人在 TM 看板、每个完成都出经验时，可在此方向扩展。

### 3.6 可选扩展：CCPM 风格 worktrees + TM Task（与方案一/三兼容）

- 在**方案一或方案三**基础上，为「多任务并行、工作区隔离」增加 **worktree 与 task_id 绑定**（不依赖任何 Issue 系统）：
  - **tm_ready** → 选任务 → **tm_task_claim(task_id)** → 创建或绑定该任务对应 worktree，在对应目录开发；
  - 完成后 **tm_task(update, completed, summary)** 沉淀经验，再释放/归档 worktree。
- 技术优势：物理隔离与并行能力，任务与经验 100% 在 TM；适合强依赖多分支/多目录的开发者或 Agent 并行流。

## 四、方案三：任务组 = 交付单元，完成时按「总-分」或「总-分-分」沉淀（与方案一组合）

> 与方案一组合使用：单任务完成只写**单经验**；任务组全部完成后，视组内任务是否**紧相关**选择 **总-分** 或 **总-分-分**，避免「纯单经验关联弱、一个经验组压扁又过紧」的问题。

### 4.1 何时用总-分、何时用总-分-分

- **任务组内任务紧相关**（如同一需求下的连续步骤、同一故障的排查与修复、同一模块的一串小改动）：**不归类**，直接按 **总-分** 写入即可——**总** 一条交付总览，**分** 为组内各单经验直接挂在总下，形成「总 → 单经验1、单经验2、…」的两层结构。
- **任务组内任务类型混杂**（如既有 UI 调整，又有逻辑调整、技术选型、配置修改等）：对组内单经验**按类型归类**，采用 **总-分-分**——**总** 为交付总览，**分（类别）** 把同类型单经验归到一类下，**分（单经验）** 仍是叶子；便于按类型、按总览、按单点多路径检索。
- **只保留单经验**：关联弱，难以从「这次交付」整体复盘；**全部压进一个经验组**：又过于紧密。总-分 / 总-分-分 在「总」下保留单经验可见性，按是否需按类型拆维选择分一层还是两层。

### 4.2 方案描述

- 把「一个需求 / 一次迭代 / 一次故障」拆成多个 **tm_task**，归到同一 **TaskGroup**（group_id）。
- 用 **tm_dependency** 标清 blocks 关系；**tm_ready** 只显示依赖已满足的子任务。
- **单任务完成**：**tm_task(update, status=completed, summary=...)** → 先写入**单条经验**（与方案一一致），单经验可带 experience_type、labels 等便于后续归类。
- **任务组 100% 完成**时，不简单压成一个经验组，而是：
  1. **归类**：对任务组内已产生的单经验按类型/主题归类（如：UI 调整、逻辑调整、技术选型、配置修改等），可依据 experience_type、labels、或人工/Agent 标注的类别。
  2. **总**：生成一条「**交付总览 / 任务组总结**」经验（父或组级），概括本次交付的背景、范围、整体结论，并引用或关联下面的「分」。
  3. **分（类别）**：按归类结果，为每个类别生成一层「**分**」（如一条「类别总结」经验，或一个结构化节点），该类别下**关联该类对应的单经验**（即任务完成时已写入的那些单经验）。
  4. **分（单经验）**：单经验本身不变，作为叶子挂到对应类别下；既可被「总」→「类别」→「单经验」的路径找到，也可单独被 tm_search 命中。

结构示意：

```
总：交付总览（一条经验，描述本次交付整体）
├── 分（类别1，如 UI 调整）→ [单经验A, 单经验B, ...]
├── 分（类别2，如 逻辑调整）→ [单经验C, 单经验D, ...]
└── 分（类别3，如 技术选型）→ [单经验E, ...]
```

### 4.3 与 team_memory Task 的结合方式

| 能力 | 用法 |
|------|------|
| **tm_task** | 子任务 create 时带 group_id；每个子任务 completed + summary → **先写单经验**（不等待组完成） |
| **tm_task_claim** | 对子任务认领；WIP 限制避免并行过多 |
| **tm_ready** | 只列「无未完成 blocking 依赖」的子任务，减少阻塞与半成品堆积 |
| **tm_dependency** | 子任务间 blocks/related，决定就绪顺序 |
| **任务组完成** | **紧相关** → 不归类，直接 **总-分**（一条总经验 + 组内单经验直接挂总下，如 tm_save_group 父+子）。**类型混杂** → 归类后 **总-分-分**（总 + 按类别组织分并关联单经验，可用 tm_save_group 或总经验 structured_data 存 类别→[exp_id]） |
| **Web 看板** | 任务组进度环、分组视图、归档；组完成可展示「沉淀为总-分或总-分-分」入口（可由人工/Agent 选择紧相关→总-分 或 归类→总-分-分） |

### 4.4 技术优势

- **依赖与就绪闭环**：tm_dependency + tm_ready 已具备；按依赖执行、减少阻塞。
- **单经验优先**：每个任务完成即产生单经验，检索与复用粒度细；不依赖「等整组完成才沉淀」。
- **总-分 / 总-分-分**：紧相关时总-分两层即可；类型混杂时总-分-分既有总览又有按类型分维，便于按类型、按总览、按单点多路径检索。
- **与现有 TaskGroup 一致**：group_id、任务组完成度、归档能力已存在；需约定「组完成时判断紧相关→总-分，否则归类→总-分-分」的流程（人工或 Agent）。

### 4.5 收益

- **可监督**：Web 看板按分组/任务组展示，进度环清晰；单任务完成即有单经验，组完成后再形成总-分或总-分-分视图。
- **可打断 / 可重启**：子任务级 in_progress + tm_message；恢复从某子任务继续；已完成的单经验不丢失。
- **可重复**：同一交付单元模版可复制；总-分-分结构可复用于迭代复盘。
- **项目助力**：按交付单元复盘、按类型检索（如「这次迭代里所有 UI 相关经验」）、按单点检索三者兼顾。

### 4.6 适用场景

- 与**方案一**组合：日常单任务用方案一；有任务组时用本节的**总-分**（紧相关）或 **总-分-分**（类型混杂）沉淀。
- **紧相关**（同需求/同故障/同模块的一串步骤）→ 总-分；**类型混杂**（UI+逻辑+技术选型等）→ 总-分-分。强调按交付单元复盘与检索，单经验独立可查、又不过于松散或过紧。

## 五、演进方向：架构管理 + 经验挂载到架构图（Case 2）

> 与 GitNexus 类似：对项目整体架构提供可视化，并把经验沉淀**挂到架构节点**上，提升按「位置」检索与推荐的质量。与当前方案一/三天然衔接。

### 5.1 目标

- **架构可视化**：管理项目架构（文件/函数/类/模块、调用关系、执行流、聚类等），在 Web 或集成环境中提供**架构图**视图。
- **经验挂载**：
  - **任务完成后**：将本次任务关联的经验（及改动的文件/符号）**同步到架构图**，挂到对应节点（如改动的 file、function、cluster）。
  - **后续改动某部分代码时**：根据当前编辑或 diff 的**架构位置**，自动或推荐把相关经验展示/挂到对应节点，使「改这里能看到这里的历史经验」，**提高项目经验沉淀的质量与命中率**。

### 5.2 与当前任务流的衔接

- **方案一 / 方案三（含总-分-分）** 的「完成即沉淀」可**预留**：
  - 单任务完成时除 `summary` 外，可记录「本次改动涉及的架构节点」（如 git diff 涉及的文件、符号或 GitNexus 的 File/Function 等）；或由后续分析任务从 commit/diff 解析出架构节点列表。
  - 单经验、以及总-分-分中的「总」「分（类别）」经验，均可增加「架构节点引用」（如 `file_path`、`symbol_id`、`cluster_name`），为「挂到架构图」与「按位置检索」打基础。
- **工作流**：单任务完成 → 单经验（现有）→ **可选**写入「经验 ↔ 架构节点」关联；任务组完成 → 总-分-分生成后，总/分类别经验也可参与架构挂载；开发时打开某文件/某符号 → 展示该位置相关经验或推荐挂载。

### 5.3 技术要点（演进时考虑）

- **架构数据来源**：可复用或对接 GitNexus 类能力（call graph、clusters、processes）；或自建轻量索引（文件/符号/依赖关系）。
- **经验 ↔ 节点**：经验与「文件/函数/模块」多对多关联；支持「任务完成时自动挂载」与「人工/Agent 后续挂载到某节点」。
- **检索增强**：tm_search / tm_solve 可接受「当前文件或符号」上下文，优先返回挂载在该位置的经验，或推荐「可挂到当前修改位置」的历史经验。

### 5.4 对方案一/三的指导

- **当前设计预留**：任务/经验的元数据或扩展字段中，可预留「affected_paths」「affected_symbols」或「architecture_node_ids」，便于未来与架构图同步。
- **不改动现有闭环**：方案一/三的「TM 唯一任务源 + 完成即沉淀」闭环不依赖架构能力；架构为**增强层**，上线后逐步接入即可。

## 六、三方案对比与选型建议

| 维度 | 方案一：TM 唯一任务源 | 方案二：TM + 外部工单（未来扩展） | 方案三：任务组 + 总-分/总-分-分（与方案一组合） |
|------|------------------------|-----------------------------------|--------------------------------------------|
| **当前版本** | ✅ 主推（与方案三组合） | ❌ 不实现 | ✅ 主推（任务组时用总-分-分） |
| **与 tm_* 对接** | 原生一致，零额外开发 | 需适配器同步/回写工单 | 单任务→单经验；组完成→归类→总-分-分 |
| **可监督** | TM Web + list | TM + 工单双端（未来） | TM Web 分组/进度环 + 总-分-分视图 |
| **可打断/可重启** | 最好（全在 TM） | 以 TM 为主 | 子任务级，组级不丢；单经验先落库 |
| **完成即沉淀** | 单任务 → 单经验 | 单任务 → 单经验 + 回写工单（未来） | 单任务→单经验；组完成：紧相关→总-分，类型混杂→归类→总-分-分 |
| **企业内网/无 GitHub** | ✅ 完全适用 | 未来可对接内网工单 | ✅ 完全适用 |
| **架构演进衔接** | 可预留 affected_paths/symbols | 同左 | 同左（单经验/总/分均可挂架构节点） |

**选型建议（当前版本）**：
- **推荐**：**方案一 + 方案三 组合**。日常以 **方案一**（TM 为唯一任务源、单任务完成即单经验）；存在任务组时，采用 **方案三**：单任务完成先写单经验，任务组全部完成后——**若组内任务紧相关**则**不归类**，直接 **总-分**（一条总 + 各单经验挂总下）；**若类型混杂**则对组内经验**归类**后 **总-分-分**（总 + 分（类别）+ 分（单经验）），兼顾关联与可检索性。
- **企业内网或无法使用 GitHub Issue**：仅用方案一+三即可，不依赖任何外部 Issue/工单。
- **未来若需与 Jira/GitLab/自建工单对接**：再扩展 **方案二**（适配器形态）。
- **多任务并行与工作区隔离**：在方案一或方案三上叠加 **worktrees + tm Task**（见 3.6），不依赖 Issue。

**架构演进（Case 2）**：方案一/三在设计上预留「任务/经验 ↔ 架构节点」的扩展点；总-分-分中的单经验、类别总结、交付总览均可参与架构挂载，后续可接入「架构可视化 + 经验挂载到架构图 + 改动时按位置推荐经验」，提升沉淀质量。
