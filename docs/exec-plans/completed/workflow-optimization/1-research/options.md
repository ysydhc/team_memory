# 开发者工作流 — 三方案选型

> 基于三视角（技术实用程度、使用者角度、项目助力角度）PK 后的最终三方案。  
> **约束**：所有方案均与 team_memory 现有 Task 能力深度结合。

## 选型结论

| 方案 | 说明 | 本期 |
|------|------|------|
| **方案一：TM 为唯一任务源** | 任务只在 TM 创建与推进；选活用 tm_ready，认领用 tm_task_claim，完成用 tm_task(update, completed) 自动沉淀经验 | ✅ 主推 |
| **方案二：GitHub 为主 + TM 执行** | TM 看板 + 外部 Issue/工单同步 | 未来扩展（企业内网/无 GitHub 不实现） |
| **方案三：任务组 = 交付单元 + 经验组** | 任务组完成时按总-分或总-分-分沉淀，紧相关不归类 | ✅ 主推 |

**PK 规则**：三个视角都强烈重合的「TM 为唯一任务源」→ 方案一；「任务组 = 交付单元 + 整组沉淀为经验组」→ 方案三。方案二因企业内网/无 GitHub 约束，当前版本不实现。

## 演进方向

- **架构管理 + 经验挂载**：TM 将引入项目架构管理能力（类似 GitNexus）；任务完成后经验同步到架构图，挂到改动涉及的架构节点；按架构位置检索与推荐。
- 方案一/三的「完成即沉淀」在设计上预留「影响的架构节点」字段，便于未来闭环。
