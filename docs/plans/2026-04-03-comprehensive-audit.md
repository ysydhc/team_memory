# TeamMemory 全面审核优化计划

**Goal:** 逐维度审核 TeamMemory 项目，发现问题并逐步修复，每个阶段先审后改。

**方法:** 每个 Phase 开始时：扫描 → 列出问题清单 → 讨论优先级 → 执行修复 → 验证

---

## Phase 1: 安全审核
- 认证/鉴权漏洞（API Key 存储、session 安全、权限绕过）
- 输入校验（SQL 注入、XSS、参数边界）
- 密钥管理（硬编码、日志泄露、env 安全）
- OWASP Top 10 对标

## Phase 2: 数据模型审核
- 索引合理性（缺失索引、冗余索引、pgvector 索引策略）
- Schema 设计（字段类型、约束、范式）
- 迁移安全（回滚能力、数据兼容性）
- 查询计划分析

## Phase 3: 性能审核
- N+1 查询、eager/lazy loading
- 连接池配置
- 缓存命中率与失效策略
- 向量搜索 + FTS 慢查询优化
- Embedding 调用频率与批处理

## Phase 4: 可靠性审核
- 错误处理完整性（是否有静默吞错误）
- LLM 调用超时/降级策略
- 数据一致性（事务边界、并发写入）
- 优雅关停（进程信号处理）

## Phase 5: 测试质量审核
- 覆盖盲区（未测路径、边界条件）
- 测试设计（mock 是否过度、fixture 是否合理）
- 集成测试可靠性（环境依赖、flaky test）
- 20 个 skip 测试的修复

## Phase 6: 可观测性审核
- 日志规范（结构化、级别、上下文）
- 关键路径指标（搜索延迟、保存延迟、缓存命中）
- 错误追踪与告警
- 调试支持（MCP IO 日志、请求链路）

## Phase 7: API 设计审核
- MCP 5 个工具的语义清晰度与参数设计
- REST API 一致性（命名、状态码、错误格式）
- 版本策略（向后兼容保证）
- 文档与实现的一致性

## Phase 8: 文档准确性审核
- CLAUDE.md 与实际代码的一致性
- docs/ 下各文档是否过时
- 代码注释准确性
- 开发者入门体验

## Phase 9: 依赖管理审核
- 过期依赖检查
- 已知安全漏洞扫描
- lock file 一致性
- 最小依赖原则（是否有不必要的包）

## Phase 10: 运维/部署审核
- Docker 配置合理性
- 迁移流程安全性
- 健康检查完整性
- 回滚策略

## Phase 11: 可扩展性审核
- 无状态设计检查
- 水平扩展瓶颈（单例、本地缓存、文件存储）
- 事件系统扩展性
- 多租户/多项目隔离

## Phase 12: 代码残留清理
- 重构后的死代码/孤立文件
- 过时的 import 和 re-export
- 无用的配置项
- TODO/FIXME 清理
