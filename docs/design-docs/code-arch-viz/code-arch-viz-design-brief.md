# 代码架构可视化方案 — 设计任务书（三视角共用）

> **目标**：设计类似 GitNexus 的代码架构可视化能力，用**最简方式**实现，并与 team_memory 项目结合（后续经验可挂载到架构节点）。

## 1. 参考：GitNexus 能力范围

- **知识图谱**：节点 File / Function / Class / Method / Process；边 CALLS / IMPORTS / EXTENDS / IMPLEMENTS / DEFINES / MEMBER_OF / STEP_IN_PROCESS。
- **能力**：codebase 概览与陈旧度、按概念 query 执行流、符号 360° 视图（调用方/被调方/所属流程）、模块聚类（clusters）、执行流步骤、影响面分析（blast radius）、Cypher 查询。
- **典型用法**：理解架构、影响分析、追 bug、重构前依赖梳理。

## 2. 项目约束

- **宿主项目**：team_memory（Python 后端、Web 前端、MCP 工具、经验库）。
- **后续演进**（见 workflow-pm-final-three-solutions）：任务完成/改动代码时，将经验挂载到「架构节点」（文件/函数/模块），按位置检索与推荐。
- **实现要求**：最简 — 可分期、可裁剪，先跑通核心价值再扩展。

## 3. 三视角分工（Composer 三 Agent）

| Agent | 视角 | 产出要求 |
|-------|------|----------|
| **A** | **技术实现最简化** | 方案优先：最少依赖、最少存储、最少算力；可接受功能裁剪；给出技术栈与实现步骤。 |
| **B** | **项目收益最大化** | 方案优先：对 TM 用户价值（任务-架构关联、经验挂载、检索增强）；可接受一定实现复杂度；给出收益与优先级。 |
| **C** | **使用最易** | 方案优先：零配置/一键、清晰心智模型、文档与引导；可接受技术折中；给出使用路径与验收标准。 |

## 4. 综合要求

- 三个方案均需：**目标用户**、**核心功能列表**、**与 TM 的集成点**、**分期建议（MVP vs 后续）**。
- 最终由主 Agent 综合 A/B/C 三方案，给出**最均衡**的一版方案（含技术选型、功能优先级、使用路径、实施步骤）。
