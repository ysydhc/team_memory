# 数据库设计决策

> 技术架构 / thinking | 为什么这样设计
> 表结构：[database-schema](database-schema.md)

## 为什么选 PostgreSQL？

| 对比项 | PostgreSQL | MySQL | SQLite |
|--------|-----------|-------|--------|
| pgvector 扩展 | 原生支持 | 不支持 | 不支持 |
| 全文搜索 (FTS) | 内置 | 基础支持 | 不支持 |
| ARRAY 类型 | 原生支持 | 不支持 | 不支持 |
| 并发性能 | MVCC，读写不冲突 | InnoDB 行级锁 | 单写锁 |

**一句话**：需要向量搜索 + 全文搜索，PostgreSQL + pgvector 是唯一能在一个数据库里同时满足的方案。

## 为什么用 Docker 运行？

- `pgvector/pgvector:pg16` 预装 pgvector 扩展，无需手动编译
- `volumes: pgdata` 数据持久化
- `healthcheck` 自动检测容器就绪

## 为什么用 UUID 主键？

| 对比项 | 自增 ID | UUID |
|--------|---------|------|
| 唯一性 | 仅单表内 | 全球唯一 |
| 可预测性 | 容易猜测 | 无法猜测 |
| 分布式友好 | 需中心化分配 | 任何地方独立生成 |

## 为什么用 ARRAY 存标签？

| 对比项 | ARRAY | 关联表 |
|--------|-------|--------|
| 查询 | 一次读取 | 需要 JOIN |
| 写入 | 直接设置整个数组 | 先删后插 |
| 适用场景 | 标签少（< 20）且无需单独管理 | 标签需独立管理 |

## 为什么用软删除？

1. **误删恢复**：用户不小心删了可以恢复
2. **数据审计**：保留历史记录
3. **延迟清理**：可定期清理已删除记录
